# Answers for Your Mentor: Data Source, Format, and Training

Use this when your mentor asks: _Where did you get the data? What format are the files? How did you train the model?_

---

## 1. Where did we get the data?

We use **two possible sources** in this project:

### Primary source (real exercise videos)

- **Name:** Real-Time Exercise Recognition Dataset
- **Where:** **Kaggle**
- **Link:** https://www.kaggle.com/datasets/riccardoriccio/real-time-exercise-recognition-dataset
- **Authors / reference:** Riccardo Riccio et al. (same as the paper we base the method on: “Real-Time Fitness Exercise Classification and Counting from Video Frames”, arXiv:2411.11548)
- **What it is:** A dataset of **exercise videos** (e.g. .mp4) for four classes: squat, push-up, shoulder press, bicep curl.
- **How we get it:** We download it with the **Kaggle API** (script: `scripts/download_dataset.py`). The user needs a one-time setup: Kaggle account → Create API Token → save `kaggle.json` in `~/.config/kaggle/` (see `docs/CREDENTIALS.md`).

So the answer: **“We downloaded the data from Kaggle: the ‘Real-Time Exercise Recognition Dataset’ by Riccardo Riccio. The dataset contains exercise videos. We run pose estimation on these videos to get the features we use for training.”**

### Other option (direct download – different format)

- We also have a script that downloads from a **direct URL** (no Kaggle): `scripts/download_real_dataset.py`.
- By default it downloads the **MM-Fit** dataset (https://mmfit.github.io/).
- That download gives **not videos or images**, but **pre-processed files**: **.npy** (NumPy arrays: e.g. accelerometer, gyroscope, 2D/3D pose) and **.csv** (labels). So in the `data/real_dataset` folder you see **.npy and .csv**, not images.
- Our **current training pipeline** in this repo is built for: **(a) videos** (we run MediaPipe and build our 22-D features) or **(b) our own pre-built sequences** (e.g. from the Kaggle videos or from demo data). We do **not** currently convert MM-Fit’s .npy/.csv into our 22-D format in the code. So for “where did we get the data for training?” the clear answer is: **Kaggle (videos)** or **synthetic demo data** (see below).

### Demo / testing (no external download)

- For testing the pipeline without downloading anything, we can generate **synthetic data** (script: `scripts/generate_demo_data.py`): random 30×22 sequences with class-specific patterns.
- So: **“We can also train on synthetic demo data generated by our script, with no external dataset, just to verify the pipeline and show baseline vs improved model.”**

## 2. What format are the files? (Why real_dataset is not images)

- **Kaggle path:** After download we have **video files** (e.g. .mp4) in a folder (e.g. `data/kaggle_exercise/`). Those are the **source**. We do **not** train on raw images; we first run **pose estimation (MediaPipe)** on each video frame, then compute a **22-dimensional feature vector** per frame (angles and normalized distances between body joints). So the **input to the model** is **sequences of 22-D vectors**, not images.

- **real_dataset folder (from direct download / MM-Fit):** The files there are **.npy** and **.csv**:
  - **.npy:** NumPy arrays (sensor data like accelerometer/gyroscope, and sometimes pose_2d / pose_3d from the MM-Fit study).
  - **.csv:** Labels or metadata.  
    So **they are not images or videos**; they are already processed data from another study. Our app’s training scripts in this repo are designed to use **either videos** (and then we create our 22-D features) **or** our own **feature CSVs / .npz** (from those videos or from demo data). We do not currently use MM-Fit’s .npy/.csv as input to our BiLSTM.

**Short answer for mentor:**  
“The **source** we use for training is either **videos from Kaggle** or **synthetic sequences** we generate. The files in **real_dataset** come from a direct-download dataset (MM-Fit) and are **pre-processed data** (.npy, .csv), not images or videos. Our model is trained on **sequences of 22-D pose features** (angles and distances per frame), not on raw images.”

## 3. How did we train the model?

- **Input to the model:** For each sample we use a **sequence of 30 consecutive frames**, each frame represented by a **22-D feature vector** (8 joint angles + 12 normalized distances + 2 vertical distances from pose). So one sample has shape **(30, 22)**.

- **Labels:** Four classes: squat, push-up, shoulder press, barbell biceps curl (names are normalized so they match the app).

- **Pipeline:**
  1. **Videos** (from Kaggle) → run **MediaPipe** on each frame → get landmarks → compute **22-D vector** per frame → save per-video **feature CSVs**.
  2. **Build sequences:** Sliding 30-frame windows over each video’s features → one sample per window → save **data_sequences.npz** (X, y, labels).
  3. **Train:** We use a **BiLSTM** (2 layers, 128 and 64 units), **BatchNorm**, **dropout**, optional **Gaussian noise augmentation**, **early stopping** on validation accuracy, and **learning-rate reduction**. We split data into train/validation/test and save the best model (by validation accuracy) and report **test accuracy**.

- **Where the code is:**
  - Feature extraction: `scripts/extract_features.py`
  - Sequence building: `scripts/create_sequence_of_features.py`
  - Training: `scripts/train_bidirectionallstm.py`
  - One full run (download Kaggle → extract → sequences → train): `scripts/run_full_pipeline.py`

**Short answer for mentor:**  
“We train a **BiLSTM** on **30-frame sequences of 22-D pose features** (angles and distances from MediaPipe). The data comes from **exercise videos** (Kaggle) or from **synthetic demo data**. We use **early stopping**, **augmentation**, and **train/val/test splits** and save the best model to `models/` for the app.”
